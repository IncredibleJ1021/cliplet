# cliplet v0.4.1 可靠性优化与发版设计

## 背景

cliplet v0.4.0 已具备文本/图片历史、全局快捷键和自动粘贴能力。本次补丁聚焦已经确认的高风险路径：错误粘贴、图片重复写盘、破坏性设置以及发布产物缺少约束。完成后发布 `v0.4.1`。

## 目标

1. 只有剪切板写入成功且目标应用确认激活后，才发送粘贴快捷键。
2. 从历史中重新选择图片时复用现有图片文件，不再写入副本。
3. 无效历史容量输入不触发裁剪；清空历史需要显式确认。
4. 发布构建失败时立即失败，不使用与 SwiftPM 配置不等价的 `swiftc` fallback。
5. 发布产物同时包含 `arm64` 和 `x86_64`，并在打包时验证版本、架构、最低系统版本和签名完整性。
6. 在自动化测试和发布 smoke check 覆盖上述行为后发布 `v0.4.1`。

## 非目标

- 不在本补丁中迁移 `UserDefaults` 历史清单或引入新的 `HistoryRepository`。
- 不增加图片缩略图缓存、总字节配额、敏感剪切板过滤、登录启动或自动更新。
- 不引入第三方依赖，不改写为 SwiftUI。
- 不在缺少 Developer ID 证书和 CI secrets 的情况下伪造签名公证；本次保留 ad-hoc 签名，并明确记录该限制。
- GitHub Release 与 Homebrew Cask 属于本次必达发布范围；npm registry 仅在当前环境已有有效发布身份时发布，否则只更新并验证 npm 包版本。

## 设计

### 1. 历史条目提升与图片去重

`ClipboardHistory` 新增按条目 ID 提升顺序的操作。该操作保留图片的文件存储 key，只更新时间和列表顺序，然后持久化元数据。

`addImageData` 在写盘前先计算大小和指纹，查找候选条目；候选命中后再比较原始字节。确认相同则直接提升旧条目，避免哈希碰撞误判和重复文件写入。新图片仍按现有原子文件路径保存。

历史面板选择已有文本或图片后调用提升操作，不再通过 `add`/`addImageData` 重建条目。

### 2. 剪切板写入与自动粘贴

剪切板写入收敛为一个可注入的小类型，显式返回成功或失败：

- 文本必须检查 `setString` 返回值。
- 图片必须同时拿到文件数据和 pasteboard type，并检查 `setData` 返回值。
- 写入失败时保留面板、播放系统提示音，不更新历史、不同步 change count、不发送 `Command-V`。

自动粘贴改为异步结果：

1. 检查辅助功能权限。
2. 检查目标应用存在且未终止。
3. 请求激活目标应用；激活请求失败则退回“仅复制”。
4. 在有限超时内检查目标是否成为前台应用。
5. 只有确认目标前台且键盘事件创建成功时才发送 `Command-V`。

任何失败都不得向未知前台应用发送按键。辅助功能未授权时保留现有授权提示；其他失败只保留已经成功写入的剪切板内容，并记录可诊断日志。

### 3. 破坏性操作保护

历史容量输入只有在文本能完整解析为 `1...200` 的整数时才应用。空值、非数字或越界值恢复为当前设置，不调用 `history.updateLimit`。

菜单栏的“Clear History”在真正删除前显示确认对话框，默认按钮为取消。用户确认后才删除文本元数据和图片文件。

### 4. 可验证的单一打包路径

`package_app.sh` 只使用 SwiftPM release 构建。SwiftPM 失败时脚本立即退出，删除直接调用 `swiftc` 的 fallback。

构建显式请求 `arm64` 和 `x86_64`，并从 SwiftPM 报告的 binary path 复制产物。组装应用后执行以下门禁：

- `CFBundleShortVersionString` 必须为三段数字版本。
- Mach-O 必须同时包含 `arm64`、`x86_64`。
- Mach-O deployment target 不得高于 macOS 13.0。
- `codesign --verify --deep --strict` 必须成功。

CI 与 Release runner 固定为 `macos-15`，避免 `macos-latest` 漂移。发布脚本验证：当前分支为 `main`、工作树包含未跟踪文件在内均为空、`HEAD == origin/main`、目标标签不存在，并把 `v0.4.1` 显式传给预检打包。

完整 XCTest 在当前仅安装 Command Line Tools 的开发机上不可用。发布脚本默认仍运行本地测试；当显式选择 GitHub CI 作为测试门禁时，必须使用 `gh` 验证当前 `HEAD` 对应的 CI workflow 已完成且结论为 success，不能跳过测试而直接打标签。

### 5. 发布流程

1. 将 `CHANGELOG.md` 与 npm 包版本更新为 `0.4.1`。
2. 在本地运行可用的 release 构建、应用包和 DMG smoke check，提交实现并推送 `main`。
3. 等待当前提交的 GitHub CI 完整通过；若本地具备 XCTest，也同时运行本地完整测试。
4. 在干净且与远端一致的 `main` 上创建并推送带注释的 `v0.4.1` 标签。
5. 等待 GitHub Release workflow 成功并确认 zip、DMG 可下载。
6. 对 release zip 计算 SHA-256，更新 Homebrew Cask 到 `0.4.1`，再次提交并推送。
7. 复核 Release、Cask 下载链接和仓库工作树状态。

若 GitHub Actions、标签推送或 Release 产物失败，停止后续发布步骤，不修改 Cask 指向不存在或未验证的资产。

## 测试策略

### Core 测试

- 提升文本条目时顺序和时间更新。
- 提升图片条目时 storage key 不变、旧文件仍存在且没有新文件。
- 再次添加相同图片时精确比较并复用文件。
- 指纹候选但字节不同的图片不会误去重。

### App 测试

- 文本和图片 pasteboard 写入失败时不更新历史、不启动自动粘贴。
- 目标为空、已终止、激活失败或激活超时时不发送按键。
- 目标成为前台后只发送一次 `Command-V`。
- 空白、非数字和越界历史容量不会改变当前 limit。

### 工程 smoke check

- shell、Node、plist 和 Swift package 语法检查。
- debug/release 构建与全部测试。
- 应用 bundle 版本、双架构、macOS 13 deployment target、签名验证。
- zip/DMG 命名与目标版本一致。

## 成功标准

- 所有新增与既有测试通过。
- 重新选择同一图片不产生新的图片文件。
- 自动粘贴无法确认目标时绝不发送全局键盘事件。
- 无效容量输入和未确认的清空操作不会删除历史。
- `v0.4.1` Release workflow 成功，zip 与 DMG 均存在；Homebrew Cask SHA-256 与 release zip 一致。
